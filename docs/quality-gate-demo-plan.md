# Pre-Arrival Quality Gate — Live Demo Plan (Frontend + Backend)

This plan shows how to demo the Quality Gate agent end-to-end with a frontend.

---

## 1. Prerequisites

| Step | What | Command / action |
|------|------|-------------------|
| 1.1 | MongoDB running | `mongod` or Docker with your `MONGODB_URI` |
| 1.2 | Seed data | `npm run -w server seed:quality-gate` (note printed `restaurantId` and `date`) |
| 1.3 | Backend | `npm run -w server dev` → http://localhost:3000 |
| 1.4 | Frontend | `npm run -w app dev` → http://localhost:5173 (or Vite port) |
| 1.5 | Env | App `.env`: `VITE_API_URL=http://localhost:3000` |

For a **demo-only** flow you can hardcode the seed `restaurantId` and `date` in the frontend so the audience doesn’t need to copy-paste.

---

## 2. Frontend Additions (What to Build)

### 2.1 New route and nav entry

- **Route:** `/quality-gate` (or `/receiving`).
- **Nav:** In `Layout.tsx`, add a link “Receiving” or “Quality Gate” next to Listings / Messages (e.g. only for restaurant role if you have one).

### 2.2 Quality Gate page (single page is enough for demo)

One page that supports:

1. **Input (demo mode)**  
   - Prefill or inputs for:
     - `restaurantId` (default: last seed value, e.g. from env or constant).
     - `date` (default: today or last seed date in `YYYY-MM-DD`).

2. **Action: “Run quality gate”**  
   - Button that:
     - Calls `POST ${API_URL}/agent/quality-gate/run?restaurantId=...&date=...`.
     - Shows loading, then:
       - `receivingBriefId`, count of `confirmationsRequested`, and any `deviationFlags` (list or badges).

3. **Action: “Load receiving brief”**  
   - Button or auto-load after run:
     - Calls `GET ${API_URL}/receiving-brief?restaurantId=...&date=...`.
     - Renders the brief using the **kitchen UI JSON** (or full `sections`):
       - One card/section per supplier with:
         - Supplier name, risk tier (LOW / MEDIUM / HIGH), confirmation status.
         - Per line: item name, qty, unit, packaging expectation, quick quality checks (bullets).

4. **Optional: “Supplier confirm” (for one order)**  
   - If there is at least one `confirmationsRequested`:
     - Dropdown or list to pick one `orderId` + `supplierId`.
     - Form: confirmQty, packSize, harvestDate, deliveryWindow, photoUrl (optional; required for HIGH).
     - Submit → `POST ${API_URL}/supplier/confirm` with JSON body.
     - On success, show “Confirmed” and optionally refresh the brief so that section shows “confirmed”.

5. **Deviation flags**  
   - If the run (or a later brief fetch) returns `deviationFlags`, show them (e.g. type, severity, suggested action).

---

## 3. API Calls (no SDK change required)

Use `fetch` and `VITE_API_URL` (or `config.API_URL`). No auth needed for demo if your backend allows it; otherwise pass your usual auth header.

```ts
const API_URL = import.meta.env.VITE_API_URL || 'http://localhost:3000';

// Run quality gate
const runRes = await fetch(
  `${API_URL}/agent/quality-gate/run?restaurantId=${restaurantId}&date=${date}`,
  { method: 'POST' }
);
const runJson = await runRes.json();
// runJson: { receivingBriefId, confirmationsRequested[], deviationFlags[] }

// Get receiving brief
const briefRes = await fetch(
  `${API_URL}/receiving-brief?restaurantId=${restaurantId}&date=${date}`
);
const brief = await briefRes.json();
// brief.sections[], brief.kitchenUiJson

// Supplier confirm (when you have a pending request)
await fetch(`${API_URL}/supplier/confirm`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    orderId, supplierId, confirmQty, packSize, harvestDate, deliveryWindow, photoUrl
  }),
});
```

## 4. Suggested UI Structure (Tailwind, match Cultivate theme)

- Reuse existing tokens: `earth-*`, `leaf-*`, `harvest-*`, `btn-primary`, cards with `bg-earth-50` / `border-earth-200`.
- **Risk badges:** LOW = neutral, MEDIUM = warning (e.g. harvest), HIGH = red/destructive.
- **Sections:** One card per `sections[]` (or per `kitchenUiJson.sections`), with supplier name and risk tier at the top, then a simple list of line items and quality checks.
- Keep the page **single-column** and scrollable for demo clarity.

---

## 5. Demo Script (step-by-step for audience)

1. **Setup**  
   “We have a restaurant, today’s orders already seeded, and the receiving brief is generated by the Quality Gate agent.”

2. **Open Receiving / Quality Gate**  
   Navigate to `/quality-gate`. Show restaurant + date (prefilled).

3. **Run the agent**  
   Click “Run quality gate”.  
   - “The backend loads today’s orders, scores risk per order, builds the brief from templates, and sends in-app confirmation requests for MEDIUM/HIGH.”

4. **Show result**  
   - “We get a brief ID, two confirmation requests (Strawberries and Mixed herbs), and no deviations yet.”

5. **Load / view brief**  
   Click “Load receiving brief” (or have it load automatically after run).  
   - Walk through one section: supplier name, risk (e.g. MEDIUM), line items, packaging expectation, quality checks.  
   - “All of this is template-based; no freeform LLM text in the brief.”

6. **Optional: Supplier confirm**  
   Pick one of the requested orders, fill the form (qty, pack size, harvest date, window), submit.  
   - “This records the confirmation and, if something is off, the backend creates deviation flags and suggested actions.”

7. **Optional: Show deviations**  
   If you triggered a deviation (e.g. confirm qty different from order), run again or reload brief and show the deviation flag and suggested action.

## 6. Optional Enhancements

- **Persist demo defaults:** Store last-used `restaurantId` and `date` in `localStorage` so re-opening the page keeps the same context.
- **Auto-run on open:** If you have a brief for today and a known restaurant, auto-call run (or load brief) when the page mounts.
- **SDK/OpenAPI:** Add `/agent/quality-gate/run`, `/receiving-brief`, and `/supplier/confirm` to the OpenAPI spec and regenerate the SDK so the app uses typed client instead of raw `fetch`.
- **Auth:** If the backend requires auth, use the same token/header as for Listings (e.g. from Auth0 and your apiProvider).

## 8. Quick Test (no frontend)

1. Seed: `npm run -w server seed:quality-gate`
2. Run: `curl -s -X POST "http://localhost:3000/agent/quality-gate/run?restaurantId=PASTE_ID&date=PASTE_DATE" | jq .`
3. Brief: `curl -s "http://localhost:3000/receiving-brief?restaurantId=PASTE_ID&date=PASTE_DATE" | jq .`
4. Confirm (use orderId/supplierId from confirmationsRequested):  
   `curl -s -X POST "http://localhost:3000/supplier/confirm" -H "Content-Type: application/json" -d '{"orderId":"...","supplierId":"...","confirmQty":"18","packSize":"12 x 1lb","harvestDate":"2026-03-01","deliveryWindow":"14:00-16:00"}' | jq .`

Use the same IDs in the frontend for the live demo.
